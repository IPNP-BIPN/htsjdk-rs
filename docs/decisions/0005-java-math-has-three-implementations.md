# 0005. Java has three incompatible math libraries, and the port must track which one each call site used

**Status:** accepted
**Date:** 2026-07-21
**Addresses:** risk R1, the direct descendant of
[broadinstitute/gatk#9384](https://github.com/broadinstitute/gatk/pull/9384)

## Measurement

809,930 points from a stratified corpus (special values, the probability range `(0,1]` that
log10 sees in every genotype likelihood, phred-scale magnitudes, a dense sweep around 1.0,
subnormals, and 40,000 pseudo-random bit patterns), generated by the pinned oracle
(Temurin 17.0.19, linux/amd64) and compared against Rust's `f64` methods.

Agreement is exact bit equality. "maxULP" is the largest observed gap between Rust and `Math`.

## What this establishes

**1. There is no single "Java math" to port against.** Three implementations coexist and
disagree pairwise:

- `java.lang.Math` — free to use a HotSpot intrinsic, so platform-specific. This is the one
  PR #9384 found differing by ~1 ULP across CPU architectures.
- `java.lang.StrictMath` — fdlibm, portable by specification, and the *furthest* from the
  other two (97.6% agreement with `Math` on `log10`).
- `org.apache.commons.math3.util.FastMath` — pure Java, therefore portable, and a third set of
  answers again.

GATK uses all three depending on the call site: `RecalDatum` calls `Math.log10`, `MathUtils`
imports `FastMath` and also calls `Math.pow`. **A ported call site must use the same one the
reference used.** Getting the function right but the library wrong is a silent 1-ULP error.

**2. `sqrt` is free.** Exactly 100% across all four implementations, because IEEE-754 mandates
a correctly-rounded square root. It needs no porting and no corpus entry beyond the assertion.

**3. The gap is always exactly 1 ULP, never worse.** No catastrophic divergence anywhere in the
corpus. That is what makes a port verifiable: a correct implementation is bit-exact, and a
wrong one is off by one in the last place, which the corpus catches immediately.

**4. Rust's libm is closest to `Math`, not to `StrictMath`.** Counterintuitive but explicable:
Rust's system libm and HotSpot's intrinsics both aim at correctly-rounded results, whereas
`StrictMath` is frozen 1990s fdlibm carrying its own rounding quirks. Anyone reaching for a
Rust fdlibm port to "get Java semantics" would be moving *away* from the target.

## Why the residual matters even though it is small

`exp` agrees 99.97% of the time. That reads like success and is not. HaplotypeCaller evaluates
`exp` on the order of billions of times per run, so a 0.03% divergence is millions of differing
values per sample, each capable of flipping a rounding boundary downstream. Bit-identity does
not tolerate a rate; it tolerates zero.

## Decision

`jmath` exposes three separate namespaces, `math`, `strict_math` and `fast_math`, and ported
code names the one the reference call site used. There is no single blessed `jmath::exp`,
because that API shape is exactly what would let a call site silently bind to the wrong
implementation.

`sqrt` maps to Rust's `f64::sqrt` in all three, asserted by test rather than assumed.

## Implementation path

Everything except `sqrt` needs a hand port. The targets are tractable: HotSpot's x86 intrinsics
for `dexp`/`dlog`/`dlog10`/`dpow` are table-plus-polynomial algorithms in the OpenJDK tree, not
opaque hardware, and `FastMath` and fdlibm are both plain readable source. The corpus is the
acceptance gate: 100% or it does not merge.

Priority follows what reaches output, not what is easiest:
`log10` and `exp` first (BQSR tables and `DuplicationMetrics.estimateLibrarySize`), then `pow`
and `log`, then the rest. `sin`/`cos`/`cbrt` are last; nothing on the current ladder calls them.

## Corpus

`crates/jmath/tests/data/jmath.csv.gz`, sha256
`4e811a9b63ee120585c38a6467c8f8a1764de148b2a8322d5cbf96a659d86a9e`, regenerated by
`tools/jmath-conformance/JMathDump.java` through `tools/oracle/run.sh`. Values are stored as
raw bit patterns, never decimal: decimal rendering discards exactly the difference being
measured.
